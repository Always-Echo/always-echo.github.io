## 一、本质原因：LLM 的认知模型天然适配文件系统

LLM 在预训练阶段消费了海量的代码仓库、文档、配置文件。它对文件路径、Markdown 结构、YAML/JSON 格式的理解是"原生"的，而不是通过 API schema 学来的。当你把上下文存成 `MEMORY.md`，LLM 读取它的准确率和理解深度，远高于读取一个结构化数据库的查询结果。

**文件是 LLM 的"母语"。**

## 二、上下文管理：文件解决了"状态持久化"的根本矛盾

LLM 本身是无状态的——每次推理都是一张白纸。Agent 系统需要解决的核心问题之一就是跨会话的状态延续。

传统方案（数据库、向量存储、KV 缓存）都面临一个共同问题：状态对 LLM 是不透明的，LLM 无法"感知"数据库里有什么，只能被动接收注入的内容。

文件系统方案的优势体现在三个维度：

- **可寻址性**：LLM 可以主动决定读哪个文件、写什么内容、何时更新。记忆管理从"被动注入"变成了"主动检索与写入"，Agent 获得了真正的自主性。
- **可审计性**：所有状态变更都是可见的 diff，人类可以直接检查、修改、回滚，对人机协作场景至关重要。
- **可组合性**：不同 Agent、不同会话可以共享同一套文件，天然支持多 Agent 协作而无需设计复杂的消息总线。

## 三、CLI 接口：为什么不是 REST API 或 SDK

这是一个微妙但关键的设计选择。CLI 相比 API/SDK 有几个独特优势：

- **零 schema 认知负担**：LLM 调用 `km search "关键词"` 和人类在终端里操作没有本质区别。而调用 REST API 需要理解 endpoint、认证方式、请求体结构、错误码——这些都是额外的认知开销，也是幻觉的来源。
- **shell 是通用粘合剂**：管道（pipe）、重定向、环境变量——Unix 哲学让 CLI 工具天然可组合。`km search "xxx" | summarize` 这样的链式调用，比设计一个支持两个系统联动的 API 简单得多。
- **人机同构**：同一个 CLI，人类可以直接在终端验证，Agent 也可以调用。这消除了"为 AI 专门设计接口"的额外成本，也让调试变得直观。
- **权限边界清晰**：CLI 工具的能力边界是显式的，比授予 Agent 一个宽泛的 API token 更容易做最小权限控制。

## 四、渐进式披露：文件系统作为上下文的"节流阀"

这是文件系统方案最容易被忽视、却极为关键的设计价值。

上下文窗口是稀缺资源，把所有信息一次性塞进去既浪费 token，又会因为"lost in the middle"效应导致注意力稀释。文件系统天然支持三层渐进式披露：

- **第一层：目录结构即摘要**。Agent 首先只需 `ls` 一下，就能获得整个知识空间的鸟瞰图，目录名、文件名本身就是语义索引，信息密度极高、token 消耗极低，Agent 在这一层完成路由决策。
- **第二层：文件头部即摘要**。大多数结构良好的文件都把最重要的信息放在开头，Agent 只读前 N 行即可判断是否值得完整加载，和 HTTP 的 HEAD 请求、数据库的索引扫描是同一个设计哲学。
- **第三层：按需全量加载**。只有在真正需要时，才把完整内容注入上下文，且只加载与当前任务相关的文件。

### 文件引用：惰性求值的知识图谱

文件里的引用（Markdown 链接、YAML 路径字段、Skill 的 `references/` 指向）本质上都是**指针而非值**。在引用被"解引用"之前，它的内容不占用任何上下文窗口——这和编程语言里惰性求值（lazy evaluation）的思想完全一致。

单个文件是节点，引用是有向边，整个文件系统构成一张知识图谱，Agent 的执行路径就是在这张图上的一次遍历。遍历策略完全由 Agent 自主决定，比预先把所有关联内容打平注入上下文，保留了更多的决策自由度。

引用还天然形成语义层次的分离，以 Skill 文档为例：

```text
// SKILL.md           ← 任务意图层：描述做什么
references/guide.md  ← 知识层：描述背景知识
scripts/run.py       ← 执行层：具体怎么做
assets/template.xlsx ← 资源层：原材料
```

每一层只在对应阶段被加载，关注点分离通过文件引用自然实现，无需任何框架层面的约定。

### 与 RAG 的本质区别

RAG 的检索触发条件是向量距离（统计性关联），文件引用的触发条件是显式的逻辑关系（结构性关联）——是文档作者在写作时就定义好的结构性关联。在需要精确、可靠的任务执行时，结构性引用的确定性远高于向量检索的模糊性。两者并不对立，但各自适合不同场景：RAG 擅长"找到我不知道在哪里的相关内容"，文件引用擅长"按照已知结构精确加载我需要的内容"。

渐进式披露让 Agent 在面对复杂、大规模知识空间时，能够以接近 O(log n) 的方式定位所需信息，而不是 O(n) 的暴力扫描。

## 五、Skill 系统的涌现：从工具调用到能力封装

Skill（如 CatPaw 的 `SKILL.md`）代表了一个更高层次的抽象——把"如何完成某类任务"的知识本身也文件化。

这本质上是在做**能力的外部化存储**。传统上，Agent 的能力边界由模型权重决定，是静态的。而 Skill 系统让能力变成了可插拔的、可版本化的、可社区共享的资产。这和软件工程里包管理器（npm、pip）的演化路径高度相似——当生态成熟到一定程度，能力的复用和分发机制自然涌现。

以 CatPaw 的 Skill 为例，这套设计体现了典型的三级懒加载结构：Agent 首先只看到 Skill 的描述字符串（一两句话）用于任务路由，匹配后才读取完整的 `SKILL.md`，执行具体步骤时才加载 `references/`、`scripts/` 等子资源。每一级的加载都由 Agent 的判断驱动，而不是系统强制注入。

## 六、更深层的系统论视角

从复杂系统的角度看，这一趋势反映了一个普遍规律：**当智能体需要与环境长期交互时，外部化存储（extended mind）比内部化存储更具演化优势**。

人类文明的发展也遵循同样的路径——从口口相传（内部记忆）到文字、书籍、互联网（外部化存储）。LLM Agent 正在经历类似的"文明化"过程：把知识、状态、能力从模型权重这个"大脑"里外化到文件系统这个"环境"里。

不同 Agent 或任务可以共享同一套底层文件，但通过不同的入口文件和引用路径，组合出完全不同的上下文视图。知识本身只存储一次，但可以被无数种方式组合消费——这让 Agent 系统获得了一种**无需中心化编排的知识复用能力**，而实现成本几乎为零。

## 结论

这一趋势不是偶然的工程选择，而是由 LLM 的认知特性、Agent 的状态管理需求、渐进式披露的上下文调度需求，以及 Unix 哲学的工程智慧共同决定的。

文件系统不只是一个存储方案，而是一个**上下文的调度框架**。文件的层级结构、引用关系、命名语义、内容组织方式，共同构成了一套隐式的"注意力路由协议"，让 Agent 在有限的上下文窗口内，以最高效的方式完成复杂任务。

预计这一范式会进一步强化，直到出现更好的解决"状态持久化"和"能力分发"问题的抽象层——但那个新抽象，大概率也会是某种形式的"文件"。
