**信息**

Skill 不是配置，不是文档，而是用自然语言写的程序。理解这一点，才能真正写好 Skill、用好 Skill，并为 Skill 生态建设提供正确的基础设施。

Skill 作为当下最火的 AI 应用形式，正在吸引不同角色、不同背景的人参与编写。这是好事，但也意味着我们需要更早地建立工程化的思维和配套工具，而不是等问题暴露之后再补救。

核心的认知转变只有一个：**把 Skill 当代码来对待**。严谨地设计，系统地测试，谨慎地管理权限，持续地监控行为。

自然语言降低了编程的门槛，但没有降低编程的要求。

## 一、一个让人豁然开朗的视角

当我们在写 Skill 的时候，我们在做什么？

表面上看，是在写一段自然语言描述，告诉 Agent 如何完成某项任务。但如果换一个角度来看——我们其实是在编程，只不过把编程语言换成了自然语言。

传统代码是写给编译器和解释器读的。Skill 是写给 Agent 读的。而 Agent 恰好能理解自然语言。所以编程语言的介质变了，但编程这件事的本质没有变：输入 → 处理逻辑 → 输出，依然是那套东西。

这个视角带来的第一个推论是：写 Skill 和写代码一样，需要严谨的工程思维。边界条件、异常处理、权限控制，在 Skill 里同样需要考虑。只不过你不是写 try/catch，而是写"如果遇到 X 情况，则执行 Y，否则报错并停止"。逻辑漏洞一样会导致行为异常。

## 二、自然语言作为编程语言：一次非连续的跃迁

编程语言的演化历史，是一部持续抽象的历史：

机器码 → 汇编 → 高级语言 → 领域特定语言（DSL）

每一次跃迁，都是让人更接近"意图"，让机器去处理"实现细节"。自然语言 Skill 是这条路的下一站，但它有一个独特之处：它不只是抽象层次更高，而是抽象的介质变了。之前每一层都还是形式语言，现在直接跳到了人类语言。这个跳跃是非连续的，不是量变而是质变。

但这也暴露了一个深层矛盾：自然语言天生是有歧义的、上下文依赖的。这些特性在人类沟通里是优点（灵活、高效），在编程里是缺点（不确定、不可预测）。所以写好 Skill 的人，其实是在用自然语言"模拟"形式语言的严谨性——这是一种反直觉的技能，需要同时具备业务理解力和逻辑严密性。

## 三、Skill 带来的三个深层变化

1. "任何人都能写"是双刃剑

门槛低了，但质量的方差也大了。传统代码有类型系统、lint、单元测试来兜底；自然语言 Skill 的"编译器"是 Agent 的理解能力，模糊的表达会被"脑补"填充，这反而是风险所在。

更重要的是，"了解业务"变成了核心竞争力。传统开发里，业务逻辑要先翻译成代码，这个翻译过程会损耗信息、引入误解。现在业务人员可以直接"编程"，中间层消失了。这是真正的范式转移。

2. 解释器是黑盒，且持续迭代

传统语言的语法是形式化的，违反了编译器直接报错。Skill 的"语法"是隐式的——是 Agent 的理解模型。这意味着同一段 Skill，在不同版本的模型上可能行为不同。

这带来一个严峻的工程问题：Skill 的可移植性和版本稳定性如何保证？ 传统代码可以锁定依赖版本，但你没法锁定 Agent 的理解方式。每次底层模型升级，都可能带来 Skill 行为的静默漂移。

3. 调试的哲学变了

传统代码出错，你看堆栈、看日志，是确定性的溯源过程。Skill 出错，你面对的是一个概率性的系统——Agent 在某个步骤"理解偏了"，但它不会告诉你它理解了什么，它只是执行了一个看起来合理的行为。

这让调试从"找到错误点"变成了"理解模型的推理路径"。需要的不是 debugger，而是某种意图对齐验证的工具。这个工具目前基本上还不存在，是个很大的空白。

## 四、角色的分化：谁是 AI 时代的程序员？

当编程门槛消失，"程序员"这个身份不会消失，而是分化为两类人：

意图表达者：懂业务、懂需求，能用自然语言清晰描述"要做什么"。他们现在可以直接产出"可执行的东西"，而不只是文档或需求说明。

系统架构者：理解 Agent 的能力边界、工具链的组合方式、安全模型，负责搭建 Skill 运行的基础设施。他们的工作是让意图表达者的产出能够安全、稳定、可预测地运行。

这两类人之间的协作方式，会和今天的"产品经理 + 工程师"模式很不一样——因为意图表达者现在可以直接产出可执行的东西，边界变得模糊，协作的接口也需要重新定义。

## 五、基建层应该提供什么

面对上述问题，AI 基础设施团队需要从以下几个方向发力：

确定性执行的锚点：自然语言的歧义性是根本问题，基建层可以提供一套"强类型"的原语——标准化的工具调用接口、明确的输入输出 Schema、以及预定义的错误处理协议。Skill 作者用自然语言描述逻辑，但关键的边界行为由这些原语来兜底，而不是完全依赖模型理解。

沙箱与权限模型：每个 Skill 在执行时应该有明确的能力边界声明——它能访问哪些工具、哪些数据、能不能写文件、能不能发网络请求。基建层强制执行这个边界，而不是靠 Skill 作者自觉描述。类比 Android 的权限系统，安装时声明，运行时校验。

执行轨迹的可观测性：Agent 执行 Skill 的每一步推理和工具调用都应该被记录下来，形成可回溯的执行日志。这不只是为了调试，也是为了审计——当 Skill 做了一个意外的操作，你能知道它是在哪一步、基于什么理解做出这个决定的。

模型版本与 Skill 的兼容性测试：每次底层模型升级，基建层应该有自动化的回归测试机制，跑一遍已有 Skill 的标准用例，提前发现行为漂移。这是传统软件里"依赖版本锁定"在 Agent 世界的等价物。

## 六、业务层应该提供什么

业务层更贴近 Skill 作者，核心是降低"写好"的门槛：

Skill 的测试框架：类比单元测试，业务层应该提供一套标准化的用例描述格式，让 Skill 作者可以写"给定这个输入，期望这个输出"的测试用例，然后有工具自动执行并给出通过率。这样 Skill 的质量就有了可量化的指标，而不是靠人工感知。

意图对齐的验证工具：Skill 作者写完之后，工具自动生成一批边界用例让 Agent 执行，然后把执行结果和预期意图做对比，标记出"模型理解和作者意图不一致"的地方。本质上是把 Skill review 的过程自动化。

Skill 的分级与认证机制：不同质量、不同风险等级的 Skill 应该有不同的标识。涉及写操作、外部调用的 Skill 需要经过更严格的审核才能发布；只读类的 Skill 可以快速上线。既不阻碍创新，又能控制风险。

最佳实践的模板库和 linter：针对常见的 Skill 编写模式，提供标准模板；同时提供一个"自然语言 linter"——检查 Skill 里是否有模糊的指令、缺失的异常处理、不明确的权限声明。这个 linter 本身也可以是个 Agent，专门做 Skill 的 code review。

## 七、一个更深层的建议：Skill 需要元数据规范

基建层和业务层之间还需要一个共同语言——一套描述 Skill 能力和约束的元数据规范。

类似 OpenAPI 之于 REST 接口，这套规范让 Skill 的能力边界、依赖工具、风险等级都是机器可读的，而不只是藏在自然语言描述里。有了这个，上面说的权限控制、兼容性测试、分级认证才能真正自动化，而不是靠人工维护。

这套规范的设计，本身就是一个值得深入探讨的工程课题。
