## 一、风险治理域核心问题

**① 风险判定效果如何检查？**

从最终目标看，***风险判定的结果会强介入 Agent 执行流程***，需要**保障风险判定的准确率**，目前整体能力还处于初级阶段，从 MVP 上线效果看需要进一步优化迭代，同时在后续的实施中需要建立起完善的效果保障机制，类似于风险治理系统对 Agent 的效果保障职责；措施上无需按照 Agent 的级别处理否则会陷入套娃循环中。

策略上推荐几种方案：

一是***产研人工抽检***，适合上线初期快速发现问题；

二是***训练师或者外包抽样质检***，在更大的范围下进行检查，适合稳定发展期，风险判定逻辑非常直接，基本不存在理解 Gap，所有人工介入的检测准确率会非常高；

三是采用***从严到宽松的判定逻辑迭代策略***，风险判定属于黑名单策略即对正常流程阻塞式，因此前期规则一定要严格，对明确的风险采用坚决的手段，对模糊的问题优先流出。

同时并不是所有场景都需要 99% 的准确效果，如何判断哪些场景需要强卡控准确率：一方面定义风险的等级，高等级风险对业务的影响更大，对准确率的要求也就越高；另一方面风险的执行动作，对核心流程有侵入的场景准确率要求也就越高。

**② 风险判定准确率如何优化？**

这涉及风险判定的逻辑，严格讲只有使用大模型的判定场景才会有准确率波动，因此这个问题也只会在 LLM 判定域考虑；大模型的效果优化手段比较聚焦：**Prompt 调优 + RAG + 微调 + 强化学习**。优先提示词优化 + RAG，通过 BadCase 驱动提示词优化。

**③ 风险判定策略迭代如何控制影响面？**

大模型判断策略目前的实现方案是用户自然语言定义风险 + 风险示例描述嵌入判定 Prompt 模板中，单规则的修改都可能会对整体效果产生影响，因此每次规则的迭代都需要引入检查机制。检查方向有以下几点：

① *检测描述是否准确严谨、示例是否明确*，可以通过另一个语义理解 SOTA 模型检测（比如 Qwen72b）；

② *检测与其他规则描述是否冲突/相似*；

③ *迭代生效前都优先通过评测检查*，强依赖评测集的准确和全面性，由于工作量和推进机制成本较高，低优先级引入；

④ *规则变更审核*，任何配置变更都需要引入，可以统一方案处理。

**④ 风险判定规则如何解决依赖数据？**

风险的判定会依赖 Agent 行为和结果日志，如何更方便的获取和理解是最大的挑战，目前关于 Trace 的记录还未引入规范，在具体实施之前需要更大范围共识；在这之前针对固定场景的依赖可以单独提需求解决。

**⑤ 风险判定效果如何应用？**

在确保准确率的前提下就可以将结果嵌入我们业务的流程中，需要与产品共同商议。具体应用方式有四种：

一是**可查**，风险事件需要能从各种维度审计效果来决策相关的产品策略；

二是**可感知**，尤其是高危风险可以做到相关干系人实时感知介入；

三是**可优化**，针对特定风险可以进行自动化的预案执行或者给出风险治理建议，提升问题处理效率；

四是**可干预**，根据相关信息、事件、分析结果等给出动作建议包括人机协同、对话实时感知、熔断兜底等。

**下阶段会推进的几件事：**

① 阶段性人工抽检驱动优化，产研共同敲定执行策略；

② 采用从严到宽松的判定逻辑迭代策略优化提示词，近期需要重新迭代一版；

③ 引入变更检测机制避免低级问题发生；

④ 定义风险标签并在质检报告体现；

⑤ 风险事件通知并引入人机协同策略维护好商家体验；

⑥ 结合 RAG 进行判定准确率提升。

## 二、风险治理与 AI Agent 产品的结合策略

**① 红线规则第一优先级。**

LLM 的问题在于输出的不确定性以及不可控性，与模型的训练预料和模型的默认提示词部分有很大关系，对于常规的基础内容风险和业务风险要第一优先级发现和治理。

**② 系统建设重要不紧急。**

风险治理系统在整个 Agent 建设中属于重要但不紧急的部分，但不代表风险治理并非不紧急，在建设前期可以用更便捷低成本的方案解决。

**③ ***先紧后松***。**

风险判定属于黑名单策略即对正常流程阻塞式，因此前期规则一定要严格，对明确的风险采用坚决的手段，对模糊的问题优先流出，可以避免准确率带来的误伤。
